# Jetpack Compose 환경에서 별지도(StarMap) 구현 아키텍처 전환

## 1. 설계 배경

별도리 프로젝트의 핵심 기능 중 하나는 **사용자의 위치와 시간에 맞는 실제 밤하늘을 보여주는 별지도 기능**입니다.
이를 위해 Android 환경에서 별자리, 천구 좌표계, 천체 위치를 렌더링하는 **StarMap 시스템을 구현해야 했습니다.**

그러나 프로젝트는 **Jetpack Compose 기반 UI 아키텍처**로 구성되어 있었고, 기존 별지도 구현 방식은 다음과 같은 문제를 가지고 있었습니다.

* 별지도 렌더링은 일반적으로 **OpenGL 기반 SurfaceView 구조**를 사용
* Compose는 **선언형 UI 구조**이며 기존 View 기반 렌더링과 생명주기 관리 방식이 다름
* 렌더링 루프, 터치 이벤트, 카메라 이동 로직이 Compose 구조와 충돌

즉, 기존 Android 방식으로 구현된 별지도 시스템을 **Jetpack Compose 환경에 직접 이식하는 것은 사실상 불가능에 가까운 상황**이었습니다.

초기에는 다음과 같은 방법을 시도하였습니다.

* OpenGL 기반 별지도 직접 구현
* 기존 Android View 기반 구조를 Compose에 이식
* 별지도 렌더링 라이브러리 검토

그러나 **Compose 생명주기와 렌더링 구조의 차이로 인해 안정적인 구현이 어려웠습니다.**

# 2. 아키텍처 전환 설계

이 문제를 해결하기 위해 별지도 시스템을 **새로운 구조로 재설계**하였습니다.

## 1) Stellarium 기반 별지도 엔진 채택

기존 모바일 렌더링 대신
**Stellarium Web 엔진을 앱 내부에서 실행하는 구조**를 선택하였습니다.

Stellarium은 실제 천문 데이터를 기반으로 한
**검증된 오픈소스 천체 시뮬레이션 엔진**입니다.

이를 활용하여 다음과 같은 기능을 구현할 수 있었습니다.

* 별자리 및 행성 시뮬레이션
* 적도좌표계 / 지평선 표시
* 천체 위치 계산
* 시간 기반 천체 이동

또한 Stellarium 엔진을 통해
**10만 개 이상의 별 데이터를 포함한 천체 시뮬레이션 환경**을 모바일 앱 내부에서 안정적으로 제공할 수 있었습니다.


## 2) WebView + Android Bridge 구조 설계

Stellarium Web 엔진을 앱 내부에서 구동하기 위해 다음 구조를 설계했습니다.

```
Jetpack Compose UI
        │
        ▼
StellariumController (Android)
        │
        ▼
WebView (Stellarium Web)
        │
        ▼
JavaScript API
```

핵심 설계는 다음과 같습니다.

* WebView 내부에서 **Stellarium Web 실행**
* Android에서 **JavaScript API를 호출하여 별지도 제어**
* Compose UI는 별지도 엔진과 **Controller 레이어로 분리**

이를 위해 **StellariumController**라는 중간 제어 계층을 구현하였습니다.

이 Controller는

* 카메라 이동
* 천체 선택
* 시야각 변경
* 좌표계 표시

등 **약 20여 개의 별지도 제어 API**를 제공하도록 설계되었습니다.


## 3) JSON 기반 천체 제어 인터페이스

별지도 조작을 코드에서 직접 호출하는 대신
**JSON 기반 명령 구조**를 설계하여 별지도 제어를 단순화했습니다.

예시 기능

* 특정 천체 선택
* 카메라 이동
* 확대 / 축소
* 좌표계 표시
* 격자 표시

이를 통해 별지도 기능을

* 교육 프로그램
* 시나리오 기반 학습
* 자동 시연

등 다양한 기능에서 **재사용 가능한 구조**로 설계할 수 있었습니다.


# 3. 적용 결과 및 인사이트

## 구현 결과

아키텍처 전환을 통해 다음과 같은 결과를 얻었습니다.

* Jetpack Compose 환경에서도 안정적인 별지도 기능 구현
* **10만 개 이상의 천체 데이터를 기반으로 한 실제 밤하늘 시뮬레이션 제공**
* 별지도 제어 기능을 API 형태로 모듈화
* 교육 프로그램 및 자동 시연 시스템과 자연스럽게 연동


## 기술적 인사이트

이 경험을 통해 다음과 같은 설계 원칙을 깨달았습니다.

### 1️⃣ 모든 기능을 직접 구현하는 것이 항상 최선은 아니다

천체 시뮬레이션은 매우 복잡한 문제이며
이미 수십 년간 검증된 오픈소스 엔진이 존재합니다.

직접 구현보다 **기존 엔진을 시스템에 맞게 통합하는 것이 더 효율적인 선택**이었습니다.


### 2️⃣ UI 아키텍처와 렌더링 엔진은 분리되어야 한다

Compose는 UI 프레임워크이고
천체 렌더링은 **별도의 엔진 영역**입니다.

이를 Controller 레이어로 분리함으로써
UI 변경에도 엔진 로직이 영향을 받지 않도록 설계할 수 있었습니다.

### 3️⃣ 아키텍처 전환 능력이 개발자의 중요한 역량이다

초기 구현 방식이 한계에 부딪혔을 때

* 구조적 원인을 분석하고
* 새로운 아키텍처를 설계하고
* 시스템 전체 구조를 재구성하는 과정

은 단순한 기능 구현을 넘어
**시스템 설계 역량을 크게 성장시킨 경험**이었습니다.


# JSON 기반 천문 교육 프로그램 실행 엔진 설계 및 구현

## 1. 설계 배경

별도리 애플리케이션은 단순히 별자리를 보여주는 기능을 넘어서
**천문 교육 콘텐츠를 인터랙티브하게 제공하는 기능**을 목표로 개발되었습니다.

예를 들어 다음과 같은 교육 시나리오가 필요했습니다.

* 백조자리 설명
* 특정 별로 카메라 이동
* 별도리 캐릭터 설명
* 적도좌표계 표시
* 다음 천체로 이동

이러한 교육 콘텐츠를 코드에 직접 작성할 경우 다음과 같은 문제가 발생합니다.

* 새로운 교육 콘텐츠 추가 시 앱 업데이트 필요
* 콘텐츠 제작자가 개발자가 아니면 제작 불가능
* 교육 시나리오 수정 시 코드 변경 필요

따라서 **교육 콘텐츠와 실행 로직을 분리하는 구조**가 필요했습니다.

이를 위해 **JSON 기반 시나리오 실행 엔진(EduEngine)** 을 설계하였습니다.


# 2. 교육 프로그램 실행 아키텍처

교육 프로그램은 다음 구조로 설계되었습니다.

```
EduProgram JSON
       │
       ▼
EduEngine (Scenario Runner)
       │
 ┌───────────────┬───────────────┐
 ▼               ▼               ▼
StellariumController   Live2DController   UI Overlay
```

핵심 아이디어는
**교육 프로그램을 JSON 스크립트로 정의하고 엔진이 이를 실행하는 구조**입니다.


## 1) JSON 기반 시나리오 구조

교육 콘텐츠는 다음과 같은 JSON 구조로 정의됩니다.

```json
{
  "steps": [
    { "action": "move_camera", "target": "Deneb" },
    { "action": "character_speak", "text": "저 별이 바로 Deneb입니다." },
    { "action": "toggle_grid", "grid": "equatorial" }
  ]
}
```

하나의 교육 프로그램은 평균 **30~80개의 Step**으로 구성되며
각 Step은 다음과 같은 행동을 정의합니다.

* 천체 이동
* 카메라 확대
* 별자리 표시
* 캐릭터 설명
* UI 로그 출력


## 2) EduEngine (Scenario Runner)

EduEngine은 JSON 시나리오를 실행하는 **스크립트 실행 엔진**입니다.

엔진의 역할

1. JSON 시나리오 로드
2. Step 순서 관리
3. Action Dispatcher 실행
4. 다음 단계 진행

```
JSON Step
   │
   ▼
Action Dispatcher
   │
   ├── StellariumController
   ├── Live2DController
   └── UI Controller
```

엔진은 **5개 이상의 Action Dispatcher**를 통해
별지도, 캐릭터, UI 시스템을 통합 제어하도록 설계되었습니다.

## 3) Compose 기반 상태 관리

Jetpack Compose 환경에서는 상태 관리가 중요합니다.

이를 위해 **EduViewModel을 통해 프로그램 상태를 관리**하도록 설계했습니다.

ViewModel 역할

* JSON 시나리오 로드
* 현재 Step 상태 관리
* UI 상태 업데이트
* EduEngine과 UI 연결

```
JSON → Engine → Controller → UI
```

# 3. 적용 결과 및 인사이트

## 구현 결과

이 구조를 통해 다음과 같은 기능을 구현할 수 있었습니다.

* JSON 파일만으로 교육 콘텐츠 제작 가능
* 별지도 + 캐릭터 + 설명이 결합된 인터랙티브 교육 구현
* **앱 업데이트 없이 교육 콘텐츠 추가 가능**
* 교육 프로그램 재사용 및 확장 용이

## 기술적 인사이트

### 1️⃣ 콘텐츠와 실행 로직은 분리되어야 한다

교육 콘텐츠는 지속적으로 수정되고 확장됩니다.

따라서 코드가 아니라
**데이터 기반(JSON) 구조로 설계하는 것이 유지보수에 유리**합니다.

### 2️⃣ 엔진 구조는 시스템 확장성을 높인다

EduEngine을 통해

* 별지도
* 캐릭터
* UI
* 교육 콘텐츠

가 서로 독립적으로 동작하도록 설계할 수 있었습니다.

### 3️⃣ 인터랙티브 교육 시스템은 게임 엔진 구조와 유사하다

Step 기반 실행 구조는
게임 이벤트 시스템 및 스크립트 엔진과 유사한 구조를 가지며
**콘텐츠 중심 애플리케이션에서 엔진 설계의 중요성**을 이해하게 되었습니다.

# 초단기 기상 데이터를 3차원 격자(Grid) 구조로 변환하여 Backend 서빙

## 1. 설계 배경

별도리 애플리케이션에서는
**천체 관측에 적합한 날씨를 추천하는 기능**을 제공하기 위해
기상청의 초단기 예보 데이터를 활용했습니다.

천체 관측은 다음과 같은 기상 조건의 영향을 크게 받습니다.

* 구름량
* 강수 여부
* 기온
* 풍속

그러나 기상청 API는 다음과 같은 구조적 한계를 가지고 있었습니다.

* 데이터가 **격자 좌표(nx, ny)** 기준으로 제공
* 시간별 데이터가 분리된 **평면 데이터 구조**
* 관측 추천 기능에서는 **공간 + 시간 정보를 동시에 활용해야 함**

따라서 데이터를 **공간(x,y)과 시간(t)을 포함한 3차원 Grid 구조**로 재구성할 필요가 있었습니다.

---

# 2. 3차원 Grid 데이터 구조 설계

기상 데이터를 효율적으로 처리하기 위해 다음과 같은 구조를 설계했습니다.

```
WeatherGrid[x][y][t]
```

기상청 초단기 예보 기준

* 공간 격자 약 **149 × 253**
* 약 **37,000개의 grid**

시간 차원을 포함하면

* 최대 **6시간 예보**

따라서 전체 데이터 규모는

**약 22만 개 이상의 Weather Cell**

을 포함하는 구조가 됩니다.

---

## 데이터 구조

```
WeatherGrid[x][y][t] = WeatherData
```

WeatherData

```
WeatherData
 ├─ sky
 ├─ precipitation
 ├─ temperature
 └─ wind
```

이를 통해 특정 위치와 시간의 기상 상태를
**O(1) 시간 복잡도로 조회**할 수 있도록 설계했습니다.

# 3. Backend 데이터 처리 구조

기상 데이터 처리 흐름

```
기상청 API
     │
     ▼
Backend 수집 모듈
     │
     ▼
3D Grid 데이터 변환
     │
     ▼
Weather Cache
     │
     ▼
관측 추천 API
```

Backend는 다음 역할을 수행합니다.

1. 기상청 API 데이터 수집
2. 평면 데이터를 **3D Grid 구조로 변환**
3. 서버 캐시에 저장
4. 관측 추천 API에서 빠른 조회

# 4. 적용 결과 및 인사이트

## 구현 결과

이 시스템을 통해 다음 기능을 구현했습니다.

* 지역별 천체 관측 추천 기능
* 시간 기반 관측 가능성 예측
* 사용자 위치 기반 관측 환경 분석

또한 데이터 구조 재설계를 통해

* **기상 데이터 조회 성능 개선**
* **추천 알고리즘 적용 용이성 향상**
* **API 응답 속도 단축**

을 달성할 수 있었습니다.

## 기술적 인사이트

### 1️⃣ 데이터 구조 설계는 시스템 성능에 직접적인 영향을 준다

단순 API 응답 구조를 그대로 사용하는 것보다
문제에 맞는 데이터 구조로 재구성하는 것이 훨씬 효율적이었습니다.


### 2️⃣ 공간 데이터는 시공간 구조로 설계해야 한다

기상 데이터는

* 공간
* 시간

두 차원을 동시에 고려해야 하는 **시공간 데이터**입니다.

따라서 Grid 기반 구조가 적합했습니다.

### 3️⃣ 백엔드 개발은 데이터 모델링이 핵심이다

효율적인 서비스 구현을 위해서는
단순 API 구현이 아니라 **데이터 구조 설계와 모델링이 중요하다**는 것을 경험했습니다.
